# ADR-024: RRF 融合后粗排层（Coarse Ranking）

- **日期**: 2026-06-23
- **状态**: 已采纳
- **决策者**: 架构评审

## 背景

当前 RAG 管线中，RRF 融合后直接调用 DashScope Rerank API（qwen3-rerank）做精排。RRF 融合将向量检索（top_k=10）和 BM25 检索（top_k=10）两路结果合并，去重后最多 20 条候选送入 Rerank API。

Ragas 端到端评估暴露了精度不足的问题（2026-06-22）：

| 指标 | 当前值 | 目标值 | 差距 |
|:---|---:|---:|:---|
| Context Precision（ragas 原生） | 0.7273 | ≥ 0.80 | -0.073 |
| Context Precision Doc（文档级） | 0.5214 | ≥ 0.80 | -0.279 |

逐题分析显示，CPdoc 集中在 0.4/0.6 两个分值——因为精确率为 2/5=0.4 和 3/5=0.6，意味着最终 top-5 结果中有 40-60% 的 chunk 来自非期望文档。根因是 RRF 融合后的 20 条候选池中有约 60% 噪声（BM25 关键词匹配噪声 + 向量语义近邻噪声），Rerank 精排虽然能在语义上区分好坏，但在高噪声比（60% 噪声）的输入下精度受限。

**粗排是企业生产级 RAG 的标准中间层**。在 RRF 融合和 Rerank 精排之间插入一层廉价的向量相似度过滤，将候选池从 ≤20 缩至 ~15 再送精排，是业界成熟的优化手段，可显著降低精排噪声比，提升 Context Precision。

## 决策

### 在 RRF 融合后、Rerank 精排前插入 CoarseRank 粗排

```
RRF Fusion（≤20 candidates）
    ↓
[CoarseRank] 向量相似度粗排 → 过滤低相似度 chunk + top_k 截断（≤15 candidates）  ← 新增
    ↓
[Rerank] DashScope Rerank API 精排 → top_k=5
```

### 粗排算法

- **算法**：query embedding 与每条 chunk embedding 的余弦相似度
- **复用性**：query embedding 在向量检索阶段已生成（`embedder.embed_query()`），零额外 Embedding API 调用
- **计算量**：L2 归一化后点积，N 次点积（N 为融合后候选数 ≤20），<1ms
- **阈值**：`COARSE_RANK_THRESHOLD=0.3`（余弦相似度 < 0.3 的 chunk 直接丢弃）
- **上限**：`COARSE_TOP_K=15`（即使全部通过阈值，最多保留 15 条送入精排）
- **下限保护**：粗排后候选 < RERANK_TOP_K（5）时，跳过精排直接返回粗排结果
- **降级策略**：任何异常 → 跳过粗排，直接传递原始融合结果到精排

### 配置项

```python
# backend/app/config.py
COARSE_RANK_ENABLED: bool = True       # 粗排开关
COARSE_RANK_THRESHOLD: float = 0.3     # 余弦相似度最小阈值
COARSE_TOP_K: int = 15                 # 粗排后最大候选数
```

### 备选方案

| 方案 | 描述 | 结论 |
|:---|:---|:---|
| A: 调大 RERANK_TOP_K | 增大精排输入量 → 增大精排输出量 → 后截取 | ❌ Rerank API 上限 20 条，已极限；增大 RERANK_TOP_K 不能解决噪声比问题 |
| B: 换更强的 Rerank 模型 | 如 Cohere Rerank v3、BGE-Reranker | ❌ 成本高（Cohere 按 token 计费）、延迟大（本地模型推理）；并未解决 Pipeline 结构缺失——精排前过滤是业界标准层 |
| C: 粗排 + 精排（选此） | CoarseRank 粗排过滤低分 chunk → 精排 | ✅ 零额外 API 成本、<1ms 延迟、降低精排噪声比、可配置阈值适配不同 KB 规模 |

## 后果

### 正面
- Context Precision（ragas 原生）从 0.7273 提升至 **0.8556**（+0.128），超过 ≥ 0.80 目标 ✅
- Context Precision Doc（文档级）从 0.5214 提升至 **0.60**（+0.079），低于预期的 0.65+，仍有提升空间
- 零额外 API 成本（复用 query embedding）
- 延迟几乎不变（<1ms 点积计算，远小于 Rerank API 的 ~200ms）
- 降低了 Rerank API 的输入 token 数（≤15 替代 ≤20，节省 ~25% Token）
- 粗排失败降级不阻断管线
- **实际验证**（2026-06-23 Ragas 28 题全量评估）：四项主指标全部达标（Faithfulness 0.8436 / AR 0.9210 / CPragas 0.8556 / CtxRecall 0.9702，均 ≥ 0.80）。详见 `backend/ragas_eval.md`

### 负面
- 多一层配置维护（THRESHOLD + TOP_K 两个阈值）
- 粗排阈值需按知识库规模和文档特征调优（大 KB 可能需要更高的 THRESHOLD）
- 新增一个 Python 模块（~50 行）+ 对应测试文件

### 风险
- **阈值过严**：`COARSE_RANK_THRESHOLD` 过高可能导致有效 chunk 被误杀。缓解：默认 0.3 保守值 + 下限保护（粗排后不足 RERANK_TOP_K 时跳过精排）
- **阈值过松**：`COARSE_RANK_THRESHOLD` 过低导致粗排形同虚设。缓解：`COARSE_TOP_K=15` 硬上限兜底
- **embedding 复用失效**：如果未来管线架构调整导致 query embedding 不再可用，粗排需额外 Embedding 调用。缓解：目前 query embedding 在向量检索阶段生成，管线传递自然可用
