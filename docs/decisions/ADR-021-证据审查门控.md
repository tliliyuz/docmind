# ADR-021: Evidence Review 门控与 Trace 可观测性

- **日期**: 2026-06-16
- **状态**: 已采纳
- **决策者**: 架构评审

## 背景

当前 Trace 系统记录了 5 个阶段（Intent → Rewrite → Retrieve → Rerank → Generate），回答了「系统做了什么」，但回答不了「为什么最终回答了？为什么拒答了？」。

现有基础设施已经具备：
- **ADR-019**：句级修辞角色过滤（`sentence_matcher.py`），通过 `detect_sentence_role()` 将句子分为 ASSERTIVE（陈述性知识）和 REFERENTIAL（引用性知识），过滤后仅保留陈述句
- **ADR-020**：三层程序级证据审计（`evidence_auditor.py`），在 LLM 输出后检查引用存在性、来源一致性和句级证据回溯

但存在两个缺口：
1. **句级过滤的分类数据没有进入 Trace**：管理员无法知道每个 chunk 被判定为陈述性还是引用性
2. **当 LLM 输出「未找到相关信息」时完全不可解释**：无法区分是 LLM 主动拒答还是证据不足

## 决策

### 新增 Evidence Review 作为正式 Pipeline 阶段

在 `filter_chunk_sentences()` + `match_sentences()` 之后、`build_prompt()` 之前，插入 Evidence Review 阶段：

```
Rerank → 句级过滤 → 证据句匹配 → Evidence Review → [门控]
  ├─ ALLOW → Prompt → LLM → Post审计 → Answer
  └─ REJECT → 直接返回"未找到相关信息"（不调 LLM）
```

### REJECT 门控

当所有 chunk 在过滤后均为空（零断言性句子）时，整体决策为 REJECT，理由 `NO_ASSERTIVE_EVIDENCE`。系统跳过 LLM 调用，直接返回「未找到相关信息」。

门控基于过滤后内容判定（而非原始内容），确保决策与 LLM 实际接收的信息一致。

### 三级 Trace 可观测性

`evidence_review` JSON 字段包含三层数据：

1. **摘要层**（卡片直接展示）：decision（ALLOW/REJECT）+ assertive/referential/rejected chunk 计数
2. **Chunk 决策层**（展开 JSON 查看）：Top 5 chunk 的角色分类（ASSERTIVE/REJECTED）+ 句子统计
3. **句子审查层**（仅 debug 模式）：逐句文本 + 角色 + 理由（本次实现默认不输出）

### Post-LLM 审计补填

`evidence_review.post_audit` 字段在 LLM 流完成后由 `chat_service` 通过 `recorder.set_post_audit()` 补填，承载 ADR-020 的三层审计结果。REJECT 路径不填充此字段。

### 模块划分

- `evidence_reviewer.py`（新模块）：PRE-LLM chunk 分类 + 门控决策
- `evidence_auditor.py`（已有）：POST-LLM 答案审计 + 置信度标注
- `sentence_matcher.py`（已有）：`filter_chunk_sentences()` 返回 `FilterStats` 供 `evidence_reviewer` 复用，避免重复切句+角色判定

### 异常降级

`review_evidence()` 内部异常时降级为 ALLOW（宁可多调一次 LLM，也不因审查模块 bug 导致用户被拒答），Trace 中记录 `status: "error"`。

## 实现

- `evidence_reviewer.py`：`EvidenceReviewResult` / `ChunkRoleDecision` / `SentenceReviewItem` 数据类 + `review_evidence()` 函数
- `sentence_matcher.py`：新增 `FilterStats` 数据类，`filter_chunk_sentences()` 返回值改为 `(str, FilterStats)`
- `trace.py`（模型）：新增 `evidence_review` JSON 列
- `trace_recorder.py`：新增 `record_evidence_review()` + `set_post_audit()` + `finish()` 增加 REJECT 推导
- `knowledge_pipeline.py`：Evidence Review 步骤 + REJECT 门控
- `chat_service.py`：REJECT 门控判断（`chat()` 函数内）+ post_audit 补填；`_generate_reject_response()` 已迁移至 `sse_stream.py`（2026-06-16 模块拆分）
- `TraceDetail.vue`：第 6 阶段卡片 + REJECT 样式
- 新增 `response_mode = "REJECT"`

## 后果

### 正面
- 填补了 Trace 可观测性中「为什么拒答」的盲区
- REJECT 门控节省了无效 LLM 调用（全引用性知识库问答场景）
- 过滤统计信息（FilterStats）复用消除了重复切句+角色判定
- post_audit 补填使 Pre/Post 两层审计在 Trace 中完整可查

### 负面
- `filter_chunk_sentences()` 返回值变更需要更新所有调用方（1 处 pipeline + 6 个测试用例）
- `traces` 表增加 JSON 列，每条 RAG Trace 增加约 1-2KB 存储

### 风险
- REJECT 门控可能过于激进：通过 `detect_sentence_role()` 默认偏向「断言性」（宁可放过不可错杀）缓解
- 门控仅检查 chunk 层面（不检查句子层面），边界模糊的句子可能被放过：可接受，因为 LLM 仍然可以通过 Prompt 中的陈述/引用知识判断框架自行判断

## 关联

- ADR-019（句级修辞过滤）：提供 `detect_sentence_role()` 和 `filter_chunk_sentences()` 基础能力
- ADR-020（三层证据审计）：Post-LLM 审计结果通过 `set_post_audit()` 补填到 `evidence_review.post_audit`
- ADR-017（单事务提交）：REJECT 路径的 assistant 消息 + Trace 在同一事务中提交
