# ADR-020: 程序级三层证据审计

- **日期**: 2026-06-15
- **状态**: 已采纳
- **决策者**: 架构评审

## 背景

v2.1 方案让 LLM 自我评估证据状态（输出 `EVIDENCE_STATUS: DIRECT/INDIRECT/NONE`），存在根本逻辑漏洞：证据状态的判断者和答案的生成者是同一个模型。如果模型已经误判"API 示例属于陈述知识"，它自然会输出 DIRECT 然后继续回答——后端看到 DIRECT 就放行，等于没有拦截。

核心认知：不让模型证明自己没有幻觉，而让系统验证每一句结论都能回溯到可审计的证据。

完整分析见 ROADMAP.md Phase 5.5 及 [RAG_PIPELINE.md](../../backend/docs/RAG_PIPELINE.md) §9。

## 决策

新增 `evidence_auditor.py` 模块，实现程序级三层证据审计：

### 第一层：引用存在性检查

正则匹配 `[来源N]` 标注，检测答案是否引用了具体来源。无引用的答案直接标记为低置信度。

### 第二层：来源一致性检查

统计引用涉及的唯一文档数：
- 1 个文档 → consistent（一致）
- 2 个文档 → acceptable（可接受）
- 3+ 个文档 → dispersed（分散，可疑）

### 第三层：句级证据回溯

jieba 分词提取关键词，在 `used_chunks` 中搜索匹配。>=2/3 关键词命中即认为有证据支撑。

### 综合置信度

综合三层结果计算置信度（high / medium / low），通过 SSE `sources` 事件的 `confidence` / `confidence_note` 字段传递给前端。

### 执行时机

审计在 LLM 流完成后、`sources` 事件构建阶段执行，不影响 SSE 流输出延迟。

### 容错策略

审计失败不阻断主流程（降级跳过）。

## 实现

- `evidence_auditor.py` 新模块（~232 行），包含 `EvidenceAuditResult` 数据类和 `audit_evidence()` 函数
- `chat_service.py` 在 LLM 流完成后调用 `audit_evidence()`（**更新**：2026-06-16 起，此调用已随 SSE 流生成逻辑迁移至 `sse_stream.py`）
- sources SSE 事件新增 `confidence` / `confidence_note` 字段
- 前端 `MessageItem.vue` 根据置信度展示警告提示

## 后果

### 正面
- 独立于 LLM 的程序级验证，能发现 LLM 自身无法察觉的幻觉
- 审计结果可解释（`confidence_note` 描述具体问题），用户可据此判断是否需要核实
- 不影响 SSE 流输出延迟（审计在流完成后执行）

### 负面
- 关键词匹配精度有限（jieba 分词 + 子串匹配），可能漏判或误判
- 不检测语义正确性，仅检查关键词可追溯性

### 风险
- LLM 未遵守 `[来源N]` 格式时退化为无引用检测
