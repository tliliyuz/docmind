# ADR-019: 句级修辞过滤 — 陈述知识 vs 引用知识区分

- **日期**: 2026-06-15
- **状态**: 已采纳
- **决策者**: 架构评审

## 背景

企业知识库 RAG 系统中，同一个 chunk 内可能同时包含陈述知识（定义、规定、流程说明）和引用知识（示例、测试数据、历史记录、API 响应示例）。LLM 无法区分文字的"写作目的"，将关键词重合等同于"可作为答案依据"，导致以下典型失败模式：

- API 文档中的 SSE 响应示例被当成制度正文回答
- 测试用例中的文本被当成功能说明回答

完整问题分析见 ROADMAP.md Phase 5.5 及 [RAG_PIPELINE.md](../../backend/docs/RAG_PIPELINE.md) §7。

## 决策

### 管线插入位置

在 RAG 管线的 Rerank 之后、Evidence Highlight 之前，插入句级修辞过滤步骤。

### 修辞角色判定

使用规则层 + 结构层双层判断每个句子的修辞角色：

- **规则层**：`_REFERENTIAL_PATTERNS` 正则匹配，识别引用知识的典型语言模式
- **结构层**：JSON / 代码块检测，识别结构化引用内容

修辞角色分为两类：
- `assertive`（陈述）：定义、规定、流程说明等可直接作为答案依据的知识
- `referential`（引用）：示例、测试数据、历史记录、API 响应示例等辅助性内容

### 过滤策略

仅保留陈述句（assertive）送入后续 Evidence Highlight 和 Prompt 组装。

核心策略："宁可放过，不可错杀"——过滤后为空时回退到原始 chunk 内容，确保不会因为过度过滤导致无内容可用。

### Prompt 模板升级

同时升级 `SYSTEM_PROMPT_TEMPLATE`，从简单的"请仅基于文档回答"升级为完整的陈述/引用知识判断框架，引导 LLM 自主区分两类知识。

## 实现

- `sentence_matcher.py`：新增 `detect_sentence_role()` 和 `filter_chunk_sentences()` 函数
- `knowledge_pipeline.py`：在 `match_sentences()` 前插入 `filter_chunk_sentences()` 调用
- `prompt_builder.py`：升级 Prompt 模板

## 替代方案

### 方案 A: 不做句级过滤，仅靠 Prompt 引导
- 优点：零代码改动
- 缺点：LLM 对引用知识的抵抗力不稳定，依赖模型能力，无法确定性保证输出质量

### 方案 B: 使用 LLM 做句级分类
- 优点：语义理解更深，可处理模糊边界
- 缺点：每个 chunk 增加一次 LLM 调用，延迟和成本显著上升；引入非确定性

### 方案 C: 规则 + 结构层过滤（当前方案）
- 优点：零额外延迟（纯正则 + 字符串操作）、确定性结果、同一 chunk 永远返回相同过滤结果
- 缺点：规则需手工维护，新场景需补充正则模式

## 后果

### 正面
- 预计消除 60-80% 的"引用知识被当成答案"问题
- 零额外延迟（纯正则 + 字符串操作）
- 确定性结果，同一 chunk 永远返回相同过滤结果

### 负面
- 规则需手工维护，新场景需补充正则模式

### 风险
- 过度过滤可能误删陈述知识，已通过"过滤后为空则回退到原始 chunk 内容"缓解
