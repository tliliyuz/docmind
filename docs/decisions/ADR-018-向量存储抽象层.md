# ADR-018: 向量存储抽象层

- **日期**: 2026-06-15（2026-06-18 更新：接口签名适配 Per-KB Collection）
- **状态**: 已采纳
- **决策者**: 架构评审

## 背景

REVIEW-FINAL-2026-06-15.md 审查发现：ChromaDB `PersistentClient` 直接绑定在代码各处（A1），多个模块直接 `import chromadb` 或依赖 `get_collection()`（A3），导致向量数据库实现与业务逻辑紧耦合，无法替换为 Milvus / Qdrant / Elasticsearch 等替代方案。

## 决策

引入 `BaseVectorStore` 抽象基类 + `ChromaVectorStore` 具体实现：

### 接口定义

只暴露三个核心操作：
- `async search(query_embeddings, n_results, kb_id, include, where=None) -> dict`
- `async add(ids, kb_id, embeddings=None, documents=None, metadatas=None) -> None`
- `async delete(kb_id, where=None) -> None`

> **2026-06-18 更新**：为支持 Per-KB Collection 策略（消除 ChromaDB metadata filter 全扫描性能瓶颈），接口新增 `kb_id` 参数。`search` 和 `add` 通过 `kb_id` 路由到对应 `kb_{kb_id}` collection。`delete(kb_id, where=None)` 在 `where=None` 时 drop 整个 KB collection（O(1)），`where={"doc_id": x}` 时在指定 collection 内按条件删除。`where` 在 `search` 中变为可选（仅用于 doc 级过滤，不再用于 KB 级隔离）。

### 不暴露 count()

理由：不同向量库 count 语义不同：
- Qdrant: `client.count(...)`
- Elasticsearch: `_index.stats()`
- Milvus: query aggregate

空库判断走数据库 `Document` 表 COUNT，比向量库 count 更稳定。

### 工厂模式

`chroma_client.py` 新增 `get_vector_store() -> BaseVectorStore` 工厂函数，返回模块级 `ChromaVectorStore` 单例。原有 `get_collection()` 保留（向后兼容），标记为新代码推荐使用 `get_vector_store()`。

### 异步线程卸载

`ChromaVectorStore` 每个方法内部使用 `asyncio.to_thread()` 将同步 ChromaDB 调用卸载到线程池，避免阻塞事件循环。

## 替代方案

### 方案 A: 保持原样
- 优点：零改动
- 缺点：无法替换向量数据库；同步 ChromaDB 调用阻塞事件循环（已违反 CLAUDE.md 异步 IO 约束）

### 方案 B: 完整 Repository 模式
- 优点：更彻底的解耦
- 缺点：过度设计。ChromaDB 返回格式与抽象层的 Retriever 紧密配合，过度抽象会损失性能

### 方案 C: 简单 ABC（当前方案）
- 优点：最小接口、渐进式迁移、向后兼容
- 缺点：`search()` 返回 ChromaDB 格式 dict（耦合点保留在 Retriever 层）

## 后果

### 正面
- 未来可插拔 Milvus / Qdrant / Elasticsearch 等向量数据库
- 所有 ChromaDB 同步调用已通过 `asyncio.to_thread()` 卸载
- `VectorRetriever` 可测试性提升（注入 mock `BaseVectorStore` 即可）

### 负面
- 增加一个抽象层和一个新文件
- `search()` 返回格式仍为 ChromaDB 特定（由 `VectorRetriever._parse_results()` 消化）

### 迁移状态
- `retriever.py`: ✅ 已迁移
- `document_service.py`: ✅ 已迁移
- `ingest/tasks.py`: ✅ 已迁移
- `ingest/delete_tasks.py`: ✅ 已迁移
- `tests/eval/eval_retrieval.py`: ✅ 已迁移
- `get_collection()`: 已移除（Per-KB Collection 迁移后不再需要，`init_chroma()` 仅创建 `PersistentClient`）

### Per-KB Collection 迁移（2026-06-18）
- 接口签名更新：`search`/`add`/`delete` 新增 `kb_id` 参数
- `ChromaVectorStore` 改为 Per-KB Collection 懒加载管理（`kb_{kb_id}`）
- KB 删除从 `delete(where={"kb_id": kb_id})` 改为 `delete(kb_id=kb_id)` drop collection（O(1)）
- 迁移脚本：`scripts/migrate_to_per_kb_collections.py`
- 性能效果：向量查询从 16.4s → ~35ms
